## Linked Architecture Elements

- AE-001: Task & Project Persistence Store — this Intent adds two coupled, additive, always-on responsibilities to the store. (1) On create/save the body-writing path (`serializeTask`, and the project-note body for milestones/parented items) writes a LIVE Obsidian `[[wikilink]]` to the item's IMMEDIATE parent on a SENTINEL-marked line (`<!-- pm:link -->`) — in ADDITION to the existing frontmatter parent refs (`parentId`/`projectId`), never replacing them, cannot be disabled. (2) A store-level content detector (`hasBodyContent` / `bodyContentLines`) strips frontmatter + any `<!-- pm:link -->`-marked line(s) and reports whether real body content survives, plus a non-blank line-count size hint. Ingestion / lazy body hydration (`stripAutoGeneratedContent`, `loadTaskBody`) must treat the sentinel-marked managed link as non-content so it neither counts toward the detector nor round-trips into the user's description. All of this rides inside the existing dirty-tracking / self-write-suppression machinery.

## Motivation

The engineer runs their vault as a portfolio co-managed with an AI collaborator, and leans on Obsidian's graph view to see how projects, milestones, and tasks connect. Today the plugin records parentage only in frontmatter (`parentId`/`projectId`) plus a bare `Project:`/`Parent:` wikilink line with no marker. Two costs follow. First, that unmarked line is indistinguishable from prose the user or AI wrote, so any content-aware tooling that tried to skip "empty" notes would either miss the managed link (double-counting it as content) or strip user prose that happens to start `Parent:`. Second, an agent walking the vault has no cheap, reliable signal for "does this note actually hold content worth reading?" — it must open and eyeball every file, and the managed auto-link makes even an otherwise-empty note look non-empty.

The underlying gap is that the managed backlink is neither reliably graph-legible NOR reliably machine-excludable, and there is no store surface that answers "has real content" for the CLI's planned `✎N` symbol. The cost of not closing it is a graph that under-connects (or tooling that miscounts) and an agent that cannot triage which notes to read.

## Desired Outcome

Every managed item, on create, carries a live Obsidian `[[wikilink]]` to its IMMEDIATE parent in its note BODY — a task links its milestone / parent task / project; a milestone links its project / parent milestone — written on a recognizable line ending in the sentinel `<!-- pm:link -->` (e.g. `Part of [[<Parent>]] <!-- pm:link -->`) at the top of the body. The link is ADDITIVE: the frontmatter `parentId`/`projectId` refs are unchanged, the body link is never a replacement, and it cannot be disabled. The wikilink targets the immediate parent, not always the project (a subtask under a parent task links the parent task; a top-level task or milestone under the project links the project). The sentinel keeps the `[[link]]` plainly visible and graph-indexable while letting tooling exclude the line.

A store-level content detector answers whether a note holds real content. `hasBodyContent(file)` strips the frontmatter and any `<!-- pm:link -->`-marked line(s), then returns true iff a non-blank line survives — so a note whose body is only the managed backlink reads as NO content, and it flips to true as soon as real prose is added. `bodyContentLines(file)` returns the count of those surviving non-blank lines (frontmatter + managed link line(s) excluded), the size hint the CLI's `✎N` symbol uses. Body hydration / ingestion continues to strip the managed link so it never leaks into the user-editable description.

## Non-Goals

- No removal of, or change to, the frontmatter parent refs (`parentId`/`projectId`) — the body backlink is purely additive.
- No opt-out / setting to disable the managed backlink — it is always-on.
- No CLI `✎N` rendering in this Intent — this ships the STORE surface (`hasBodyContent`/`bodyContentLines`); the symbol rendering that consumes it is a separate CLI concern and is not built here (so no AE-011 UI-primitive change).
- No change to the frontmatter schema — the backlink lives in the body, marked by an HTML-comment sentinel; no new frontmatter key.
- No backfill pass that rewrites the sentinel into every existing file up front — the managed link is written write-through on the next create/save, consistent with the existing body-writing path.
- No counting of blank lines as content — the size hint counts non-blank surviving lines only.

## Outcome Verification

On `insertTask(project, child, milestoneId)` the child's task file body carries a `<!-- pm:link -->`-marked line whose `[[wikilink]]` targets the milestone note, while the frontmatter `parentId` still equals the milestone id (R37). `hasBodyContent(file)` returns false for a task whose body is only the managed backlink and true once real prose is appended (R38). `bodyContentLines(file)` returns exactly the count of non-blank prose lines, excluding the frontmatter and the managed link line (R39). A task inserted directly under the project links the project note; a subtask under a parent task links the PARENT TASK (not the project); a milestone under the project links the project note (R40). These are the red-first outcome tests R37 (additive sentinel backlink), R38 (content detector excludes the auto-link), R39 (size hint excludes frontmatter + link line), and R40 (immediate-parent target) in `src/intention.test.ts`. `outcome_verification_grandfathered: false`.
