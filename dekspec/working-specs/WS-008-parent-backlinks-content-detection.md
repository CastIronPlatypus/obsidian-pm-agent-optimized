## Related Architecture Elements

## Governing ADRs

- none

## What This Does

On create/save, the store writes a LIVE Obsidian `[[wikilink]]` to an item's IMMEDIATE parent into the note BODY on a line ending in the sentinel `<!-- pm:link -->` (e.g. `Part of [[<Parent>]] <!-- pm:link -->`) — a task links its milestone / parent task / project; a milestone links its project / parent milestone. The link is ADDITIVE to the frontmatter `parentId`/`projectId` refs (never a replacement) and always-on. A store-level content detector answers whether a note holds real content: `hasBodyContent(file)` strips the frontmatter and any `<!-- pm:link -->`-marked line(s) and returns true iff a non-blank line survives; `bodyContentLines(file)` returns the count of those surviving non-blank lines (the CLI `✎N` size hint). Body hydration / ingestion continues to strip the managed link so it never leaks into the user-editable description or counts as content.

**Mechanism:** This component writes a sentinel-marked immediate-parent backlink on create/save and exposes a content detector that excludes frontmatter + sentinel-marked line(s) from "real content."

## What This Does NOT Do

- **No frontmatter change:** does not add or alter any frontmatter key; the backlink lives in the body, and the frontmatter `parentId`/`projectId` refs are untouched (the body link is additive).
- Does not add an opt-out — the managed backlink is always-on.
- Does not render the CLI `✎N` symbol — it ships the store surface only (no UI-primitive change).
- Does not count blank lines as content — the size hint counts non-blank surviving lines only.
- Does not backfill the sentinel into existing files up front — it is written write-through on the next create/save.

## Interfaces

### Data Interfaces

| Interface | Direction | Type / Shape / Dtype | Source or Consumer | Guarantees |
|-----------|-----------|----------------------|--------------------|------------|
| Sentinel-marked parent backlink | out | body line ending `<!-- pm:link -->` containing `[[<parentBasename>]]` | `serializeTask` / project-note body → vault file | written on create/save; targets the IMMEDIATE parent; additive to frontmatter `parentId`/`projectId`; `[[link]]` stays graph-indexable (R37, R40) |
| `hasBodyContent(file)` | out | `Promise<boolean> \| boolean` | CLI `✎` consumer / store | true iff a non-blank line survives stripping frontmatter + `<!-- pm:link -->`-marked line(s); a note that is only the managed link → false; real prose → true (R38) |
| `bodyContentLines(file)` | out | `Promise<number> \| number` | CLI `✎N` size hint | count of non-blank body lines surviving the strip (frontmatter + managed link line(s) excluded) (R39) |
| Managed-link exclusion in hydration | in | body text | `stripAutoGeneratedContent` / `loadTaskBody` | the `<!-- pm:link -->`-marked line is stripped from the user description and does not round-trip on save |

### Dependencies

| Dependency | Interface | Failure behavior |

## Domain Constraints

| Constraint | Value | Scope | Rationale |
|------------|-------|-------|-----------|
| Additive | frontmatter `parentId`/`projectId` unchanged | all-IBs | the body backlink never replaces the frontmatter ref (INT-021 non-goal) |
| Always-on | no opt-out for the managed backlink | all-IBs | the graph must stay connected (INT-021 non-goal) |
| Exclude | the `<!-- pm:link -->`-marked line from content + description | all-IBs | else the managed link double-counts as content / leaks into the description |
| Non-blank only | the size hint counts non-blank surviving lines | all-IBs | blank lines are not content (R39) |
| Immediate parent | the wikilink targets the IMMEDIATE parent, not always the project | all-IBs | correct graph nesting (R40) |

## Business Rules

1. **general** On create/save, the store writes a live `[[wikilink]]` to the item's IMMEDIATE parent into the note body on a line ending in the sentinel `<!-- pm:link -->`; this is additive to the frontmatter `parentId`/`projectId` and cannot be disabled. (R37)
2. **general** The content detector `hasBodyContent(file)` strips the frontmatter and any `<!-- pm:link -->`-marked line(s), then returns true iff a non-blank line survives — a note whose body is only the managed backlink reads as no content; adding real prose flips it to true. (R38)
3. **general** The size hint `bodyContentLines(file)` returns the count of non-blank body lines surviving the same strip; the frontmatter and the managed link line(s) are excluded from the count. (R39)
4. **general** The managed backlink targets the IMMEDIATE parent: a task directly under the project links the project note; a subtask under a parent task links the PARENT TASK; a milestone under the project links the project note. (R40)
5. **general** Body hydration / ingestion (`stripAutoGeneratedContent` / `loadTaskBody`) strips the `<!-- pm:link -->`-marked line so it never enters the user-editable description or round-trips on save. (R37/R38 consistency)

## Failure Behavior

| Failure | Detection | Assertion type | Behavior | Recovery |
|---------|-----------|---------------|----------|----------|
| Sentinel suppresses graph indexing of the `[[link]]` | build-time manual check in Obsidian | assert | move the marker (trailing sentinel / managed heading) rather than wrap the link, keeping the `[[link]]` indexable | implementer relocates the marker |
| Note has no frontmatter | `parseFrontmatter` returns whole content as body | assert | the detector applies to the whole body; a managed-link-only body still reads as no content | none needed |
| A user line coincidentally ends with `<!-- pm:link -->` | sentinel match | assert | treated as a managed line (excluded) — acceptable: the sentinel is store-managed and unlikely in hand prose | user edits the line |
