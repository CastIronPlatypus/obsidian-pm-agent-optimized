# Implementation Brief: Parent backlinks in body + content-aware detection

**Spec:** `dekspec/working-specs/WS-008-parent-backlinks-content-detection.md`
**Intent:** `dekspec/intents/INT-021-parent-backlinks-content-detection.md`
**Source AEs:** AE-001
**Depends on:** none
**Production gate:** none
**Status:** ACCEPTED

## Goal

Managed items self-connect in the Obsidian graph and are machine-detectable as empty/non-empty: the body-writing path writes a live `[[wikilink]]` to the item's IMMEDIATE parent on a `<!-- pm:link -->`-marked line (additive to the frontmatter parent refs, always-on); a store-level content detector (`hasBodyContent` / `bodyContentLines`) strips frontmatter + sentinel line(s) and reports/counts the surviving non-blank remainder — proven green by R37–R40 in `src/intention.test.ts`.

## Out of Scope

- Any change to the frontmatter parent refs (`parentId`/`projectId`) — the body backlink is additive.
- An opt-out for the managed backlink (it is always-on).
- CLI `✎N` symbol rendering (this ships the store surface only; no UI-primitive change).
- Any new frontmatter key; a backfill pass rewriting existing files up front; counting blank lines as content.

## Constraints & Decisions

- **Sentinel:** the managed backlink line ends in `<!-- pm:link -->`. Write and detect share one sentinel constant/helper (`YamlParser.ts`) so they never drift.
- **Additive:** the frontmatter `parentId`/`projectId` refs are unchanged; the body link is an addition, never a replacement; always-on (R37).
- **Detection rule:** strip frontmatter + any `<!-- pm:link -->`-marked line(s); `hasBodyContent` = a non-blank line survives (R38); `bodyContentLines` = count of non-blank survivors (R39).
- **Immediate-parent target:** task-under-project → project note; task-under-task → parent task; milestone-under-project → project note (R40) — reuse existing resolution.
- **Store-side only:** content detection is a store surface for the CLI `✎N` consumer; no AE-011 UI-primitive change.

## Test plan

- **R37 (additive sentinel backlink):** `createProject('Roadmap','Projects')`; `insertTask` a milestone, then `insertTask` a child with `parentId = milestone.id`; assert the child file frontmatter `parentId === milestone.id` AND the child body has a `<!-- pm:link -->`-marked line whose `[[wikilink]]` targets the milestone basename.
- **R38 (detector excludes the auto-link):** insert a task with no description; `hasBodyContent(file)` → false; `vault.process` to append real prose; `hasBodyContent(file)` → true.
- **R39 (size hint excludes frontmatter + link line):** hand-author a task file with frontmatter + a `<!-- pm:link -->` line + exactly 3 prose lines; `bodyContentLines(file)` → 3 (not 4).
- **R40 (immediate-parent target):** insert a top-level task (→ project), a subtask under it (→ parent task, NOT project), and a milestone under the project (→ project); assert each managed-link line's wikilink target.
- **Build-time check:** in a real Obsidian vault, confirm the `<!-- pm:link -->` sentinel does not suppress graph indexing of the `[[link]]`; if it does, relocate the marker (trailing/own-line sentinel or a managed heading) keeping the `[[link]]` indexable.
- **Guard:** full `pnpm test` stays green; R1–R36 unaffected (R37–R40 are additive; the detector + sentinel are new surfaces).
